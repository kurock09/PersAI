# ADR-081: Unified user Files architecture

## Status

Accepted

## Date

2026-05-02

## Relates to

ADR-060, ADR-072, ADR-077, ADR-078, ADR-079

## Context

PersAI currently has several implementation-level file concepts that leak into runtime and model-facing behavior:

- chat media attachments persisted as `AssistantChatMessageAttachment` rows with `attachmentId` and storage `objectKey`
- runtime-generated media artifacts represented as turn-local `RuntimeOutputArtifact.artifactId`
- assistant workspace and sandbox files persisted as `AssistantFile` rows with `fileRef`
- sandbox relative paths used inside isolated job workspaces
- object-storage paths used by media, runtime output, sandbox persistence, and knowledge uploads
- knowledge source identifiers and retrieval references used by the Knowledge plane

This split is understandable internally, but it creates a poor product and model contract. A user does not care whether a file came from upload, assistant generation, sandbox output, or prior chat delivery. The user sees files. The assistant should also reason over one product concept: files it can find, read when policy allows, create, edit, and send.

The current shape creates predictable failures:

- an uploaded chat attachment may have an `attachmentId` before a durable `fileRef` is available to the runtime/model
- generated artifacts have `artifactId` values that are valid only within the current runtime turn
- `files.send` accepts both durable `fileRef` and current-turn `artifactId`, but model-visible artifact metadata is intentionally sanitized in some tool-result paths
- `objectKey` is an internal storage locator but can appear close enough to model/runtime payloads that it becomes tempting to use as a selector
- `files.search` searches `AssistantFile`, while users may expect it to find any visible chat file

PersAI is pre-production for this surface. We should not preserve this partial contract or introduce compatibility layers around it. The right fix is a clean target architecture.

## Decision

PersAI will expose a single product-level Files architecture.

`AssistantFile` is the canonical durable file registry for every file that can be visible or reusable by the user or assistant:

- user-uploaded chat attachments
- assistant-generated media and documents
- files created or edited through sandbox-backed `files`, `exec`, or `shell` work
- files delivered as chat attachments by assistant tools
- future imported files that are meant to be user-visible or assistant-reusable

Every user-visible or assistant-reusable file receives a durable `fileRef` immediately at persistence time. File registration is not lazy and is not dependent on a later runtime hydration pass.

The product-facing selector contract is one stable identifier:

- `fileRef`

The ordinary model-facing selector contract is alias-first and runtime-owned:

- human-readable working-file aliases such as `current image #1`, `previous attachment #1`, and `last generated image`

The following identifiers are internal implementation details and must not be primary model-facing selectors:

- `attachmentId`
- `artifactId`
- `objectKey`
- storage path / `storagePath`
- raw sandbox absolute path
- knowledge source id
- retrieval reference id

`attachmentId` may remain a UI/API implementation detail for downloading or rendering a specific chat message attachment, but it is not the assistant file handle. `artifactId` is removed from the target model/tool contract; generated outputs must be registered as `AssistantFile` rows immediately and resolved inside runtime to canonical `fileRef`. `objectKey` remains storage-internal and is not a user or model selector.

Knowledge remains a separate product plane. Knowledge documents are not ordinary Files by default. A source file may have internal metadata linking it to a knowledge source, but the user-facing Knowledge Base and the user-facing Files list remain distinct concepts:

- Files: visible/reusable user and assistant files
- Knowledge: indexed source material for retrieval

If a user uploads a document to Knowledge, it belongs to the Knowledge product surface. If the same binary is also intentionally exposed as a reusable user file, it must have a separate `AssistantFile` registration and a clear provenance link. The assistant should not confuse retrieval references with sendable file handles.

## Target model

### Canonical file record

`AssistantFile` is the durable authority. It should carry enough product metadata to power both runtime and UI:

- `fileRef` (`AssistantFile.id`)
- `assistantId`
- `workspaceId`
- origin/provenance: uploaded by user, generated by assistant, sandbox-created, delivered attachment, imported
- current storage object key
- display name
- MIME type
- size and logical size
- checksum when available
- optional source message id / source attachment id / source tool code / sandbox job id
- created/updated timestamps
- lifecycle state if needed later (`ready`, `deleted`, `blocked`, etc.)

The exact schema can evolve, but the authority must stay singular: a user-visible reusable file is an `AssistantFile`.

### Runtime contract

Runtime inbound attachments should carry `fileRef` whenever they represent a persisted file. The runtime should not need to mint file refs during prompt hydration.

Tool output contracts should preserve the canonical file identity for created or generated files, but model-visible tool-result JSON must not rely on raw `fileRef` as the reusable selector. A generated image/video/audio/document is a file as soon as the tool result exists, not an ephemeral artifact that later may become an attachment.

The `files` tool should operate on alias, path, and query at the model-facing boundary, resolving to canonical `fileRef` inside runtime:

- `files.list`
- `files.search`
- `files.get`
- `files.read`
- `files.write`
- `files.write_and_send`
- `files.edit`
- `files.delete`
- `files.send`

`files.send` sends files by working-file alias or resolved query/path. It does not accept `artifactId` in the target contract.

Sandbox-backed tools may use relative paths inside the isolated workspace, but any persisted output returned to runtime becomes an `AssistantFile` with a `fileRef`.

### API contract

The public product API should expose a Files surface for assistant-scoped files:

- list files
- search files
- read metadata
- download file
- delete/archive file
- optionally rename/update display metadata

Chat attachment APIs may continue to serve chat message rendering and downloads, but should not become a second user file system. Chat attachments that are visible/reusable must point at an `AssistantFile`.

### Web/product UI

The user should see one assistant Files list with filters by provenance, not separate concepts for storage backend:

- Uploaded
- Created by assistant
- Generated media
- Chat attachments
- Sandbox-created

These are filters/badges, not separate systems.

The UI must not expose `objectKey`, raw sandbox paths, or implementation ids as normal product concepts. The durable handle can exist for support/debug copy actions, but the ordinary user model is name, type, origin, date, and actions.

## Non-goals

- No legacy compatibility mode for `artifactId`-based model calls.
- No user-facing distinction between media storage files and sandbox files.
- No raw object-storage path selectors in model-facing tools.
- No raw sandbox absolute paths in product UI.
- No merging Knowledge retrieval references into Files.
- No preserving old OpenClaw file/workspace concepts as active architecture.

## Implementation requirements

The implementation must be direct and clean:

1. Register every visible/reusable file in `AssistantFile` at the moment it is persisted.
2. Make generated media tools return `fileRef`-backed file outputs.
3. Remove `artifactId` as a model-facing send selector.
4. Ensure chat attachments link to their canonical file record.
5. Ensure runtime prompt hydration shows the assistant runtime-owned human aliases instead of raw `fileRef`.
6. Keep `objectKey` internal to storage, delivery, and signed-download boundaries.
7. Keep Knowledge source ids and retrieval references out of the Files send/read contract.
8. Add tests that prove uploads, generated artifacts, sandbox outputs, and delivered attachments all appear in Files and can be sent by alias while runtime resolves them to canonical `fileRef`.

## Execution plan

This ADR should be implemented in large, complete slices. Do not split into tiny compatibility steps, and do not add legacy selector fallbacks. Each slice should leave the product closer to the final contract and should remove obsolete behavior when replacing it.

### Slice 1 — Canonical Files authority in API/data model

Goal: make `AssistantFile` the durable product authority for user-visible files at the API boundary.

Status: implemented 2026-05-02 for chat/upload attachment origins. Chat attachments now link to `AssistantFile` through `assistant_file_id`, upload/inbound/delivery paths register canonical rows immediately, API/runtime attachment responses can carry `fileRef`, and the first assistant-scoped Files read APIs exist for list/search, metadata, download, rename/update metadata, and registry-row delete/archive semantics. Live polish on 2026-05-02 fixed refresh-time download loss for generated delivery attachments by persisting existing runtime-generated `fileRef` values back to chat attachment rows before history reloads depend on them.

Scope:

- extend the data model where needed so chat attachments can link to canonical `AssistantFile` rows
- register a file immediately when a user upload is persisted
- ensure every chat attachment that represents a reusable file has a `fileRef`
- add assistant-scoped Files API read surfaces for Settings UI and runtime use: list, search, metadata, download, delete/archive, and optional rename/update metadata
- keep `attachmentId` only for message-rendering internals and existing download mechanics, not as a model-facing file handle

Acceptance:

- uploading a PDF/image/audio/video in chat creates both the chat attachment and the canonical `AssistantFile` row in the same logical flow
- API responses needed by chat/runtime can include `fileRef`
- `objectKey` remains internal and is not exposed as a normal selector
- tests cover staged web uploads, direct chat uploads, Telegram inbound media where applicable, and file listing/search by `fileRef`/name

Out of scope:

- generated media tool output changes
- sandbox execution changes unless needed to align shared types
- web Files UI

### Slice 2 — Runtime/generated outputs become Files

Goal: remove turn-local generated-file semantics from the model-facing runtime contract.

Status: implemented 2026-05-02 for generated media outputs. Runtime image generation, image edit, video generation, and TTS now register canonical `AssistantFile` rows immediately and return durable `fileRef` metadata with generated artifacts. Model-visible tool results keep `fileRef` while hiding storage/delivery internals, `files.send` no longer accepts `artifactIds`, and same-turn generated files are delivered/reused through `fileRef`. API media delivery links chat attachments to the canonical generated file instead of creating a duplicate registry row when runtime already supplies `fileRef`.

Scope:

- make image/video/TTS/document-producing tools persist outputs as canonical Files immediately
- replace model/tool-result generated-output selectors with durable `fileRef`
- remove `artifactId` as a model-facing selector for `files.send`
- keep delivery metadata internal so API can persist chat attachments from canonical Files
- update runtime stream/done events so delivered/generated files can be reconciled through `fileRef`

Acceptance:

- a generated image/video/audio/document is reusable by `fileRef` in the same turn and later turns
- the model never needs `artifactId` to send or reference a generated file
- `files.send` can send generated outputs using `fileRef`
- tests cover generate -> send, generate -> later search/read/send, and delivery persistence

Out of scope:

- Files Settings UI
- Knowledge document linking

### Slice 3 — Runtime Files contract and Skill working-file behavior

Goal: make `files` and Skill-assisted work use one stable file selector across uploads, generated outputs, and sandbox files.

Status: implemented 2026-05-02 for canonical runtime Files continuity, then corrected 2026-05-07 for prompt/tool hygiene. Chat attachments/generated outputs/sandbox files still converge on canonical `AssistantFile` + `fileRef`, but prompt hydration no longer prints raw `fileRef` into conversational history. Runtime now exposes working files to the model through runtime-owned aliases, and `files`/`image_edit`/`video_generate` resolve those aliases server-side back to canonical `fileRef`. Focused tests cover uploaded PDF query -> read with required file mount, ambiguous query choices, sandbox-created file send, generated file search/read/send continuity, alias-based generated/current-file reuse, and prompt-hydration hygiene. Knowledge ingestion remains separate.

Scope:

- update `RuntimeAttachmentRef`, `RuntimeFileRef`, `RuntimeFilesToolResult`, and related contracts so runtime can carry canonical `fileRef` plus runtime-owned aliases
- update prompt hydration so chat files are presented as working files with aliases, not raw `fileRef`
- update `files.list/search/get/read/write/write_and_send/edit/delete/send` behavior and schemas to remove `artifactIds`
- ensure sandbox outputs continue to return canonical `fileRef`
- ensure Skills can find/read current and prior chat files through the same Files registry

Acceptance:

- a Skill such as `ГИП проекта` can work with a PDF ТЗ uploaded in chat by finding/reading it as a file
- `files.search` searches the unified user-visible assistant Files registry, not only sandbox outputs
- `files.send` accepts alias/query/path only at the model-facing boundary
- ambiguous query behavior returns clear choices instead of auto-picking by hidden raw selector
- tests cover uploaded PDF -> Skill turn -> files search/read; sandbox-created file -> send; generated file -> search/read/send

Out of scope:

- public Knowledge ingestion changes
- a folder-first user file manager

### Slice 4 — Assistant Settings Files UI and chat attachment projection

Goal: give the user one visible Files surface without introducing a separate top-level app route.

Status: implemented 2026-05-02 for the first product Files surface. Assistant Settings now includes a compact mobile-first Files section with search/refresh, quiet collapsible groups, provenance badges, inline scroll for large lists, and Download/Rename/Delete actions backed by canonical `fileRef`. Backend Files responses own the product grouping through `fileBucket`, `cleanupEligible`, and `cleanupReason`, so the web UI can group media, assistant-created files, and user files without guessing from filenames; cache/history rows are excluded from the main useful list and surfaced only through cleanup. Chat attachment cards now use the canonical Files download/open route whenever `fileRef` exists, so chat cards and settings rows point to the same File without exposing `objectKey`, raw sandbox paths, or storage internals. Live polish on 2026-05-02 made the mobile Files list quieter: all useful groups are collapsed by default, ordered media-first, and row actions live on the filename line to reduce row height. Live polish on 2026-05-02 removed the generic Files `Open` action from the settings list because raw file opens are unreliable in Capacitor; image/video Files keep a quiet in-app `Preview` action instead. Live polish on 2026-05-02 also added quiet deleted-file projection: when a Files row is deleted, linked chat attachments render disabled metadata with `файл удалён` rather than a broken download state. The same polish keeps delivered-file downloads in the attachment card by stripping/downgrading model-written markdown links to delivered filenames. Mobile download polish on 2026-05-02 fixed Cyrillic text/markdown content after Capacitor/Android downloads by declaring UTF-8 at the API boundary and adding a UTF-8 BOM for forced text downloads where legacy Android viewers need an in-file encoding hint. Cache cleanup is intentionally narrow and backend-owned: cleanup-eligible history/cache artifacts such as transient uploaded voice/audio cache are removed, while user files, assistant-created/generated files, media uploads, Knowledge, and non-deleted working files remain. Knowledge remains separate.

Scope:

- add a Files section inside Assistant Settings
- list all assistant Files with provenance filters/badges: uploaded, generated, created by assistant, chat attachment, sandbox-created
- expose user actions appropriate for the current product: download, delete/archive, rename if supported, in-app preview for image/video files, and later add-to-Knowledge if separately designed
- update chat attachment cards so they are projections of canonical Files when a `fileRef` exists
- avoid exposing `objectKey`, raw sandbox paths, or implementation ids as normal UI concepts

Acceptance:

- files uploaded in chat appear in Assistant Settings Files
- assistant-generated and sandbox-created files appear in the same list
- chat attachment cards and the Files settings row refer to the same canonical file
- user can download through the product API and preview image/video files in-app without seeing storage internals
- UI copy treats provenance as metadata, not as different storage systems

Out of scope:

- a separate `/files` route
- merging Knowledge Base into Files
- multi-user sharing/collaboration semantics

### Slice 5 — Cleanup, contract hardening, and full verification

Goal: remove obsolete selectors and prove the final architecture is the only active path.

Status: implemented 2026-05-02 for final cleanup/hardening. The product Files API response now exposes only `fileRef`, provenance, display filename, MIME/size metadata, and created time; it no longer returns storage-derived `relativePath` or `sha256`. Web chat attachment cards no longer fall back to the old attachment download route and only create open/download URLs for canonical `fileRef` files. The obsolete web `/api/attachment/:id` proxy, API `GET /assistant/attachment/:attachmentId` route, and auth allowlist entry were removed from the active path. The remaining `artifactId`, `objectKey`, and relative-path usage is internal runtime/API storage and sandbox execution state, not model/product-facing selector truth.

Scope:

- remove model-facing `artifactId` send paths and any lazy fileRef creation that survives earlier slices
- remove or hide model-visible `objectKey` and storage path leakage
- tighten generated OpenAPI/runtime contracts around `fileRef`
- update tool catalog guidance so the model sees one Files concept
- update docs and tests to match final truth
- run full verification across API/runtime/web and focused live smoke where useful

Acceptance:

- no model-facing schema asks for `artifactId` to send a file
- no model-facing prompt/tool result requires `attachmentId` or `objectKey` to operate on files
- all visible/reusable file origins are represented in `AssistantFile`
- API/runtime/web tests prove upload/generated/sandbox/chat attachment paths converge on `fileRef`
- ADR-081 implementation notes are reflected in `ARCHITECTURE`, `API-BOUNDARY`, `DATA-MODEL`, `TEST-PLAN`, `CHANGELOG`, and `SESSION-HANDOFF`

Out of scope:

- preserving compatibility with partial pre-production selector behavior
- OpenClaw workspace/file semantics

## Next-session start rule

When starting implementation work for ADR-081, begin with the first incomplete slice in the order above. Do not skip ahead to UI until the canonical file authority and immediate `fileRef` creation are in place. If code and ADR-081 disagree, reconcile the docs and implementation plan before changing runtime behavior.

The first implementation session should start with Slice 1 and should identify:

- the current `AssistantFile` schema gap for linking chat attachments
- the upload/staging paths that must register `AssistantFile` immediately
- the API response shapes that need to carry `fileRef` for chat/runtime/UI
- focused tests proving an uploaded chat PDF is already a canonical File before runtime hydration

## Final acceptance criteria

ADR-081 is complete only when all of the following are true:

- every user-visible or assistant-reusable file origin creates an `AssistantFile` row immediately
- the model-facing selector for files is a runtime-owned human alias, while canonical file identity remains `fileRef`
- generated outputs no longer require or expose turn-local `artifactId` for send/reuse
- chat attachments are projections of canonical Files where reusable
- Assistant Settings contains the user-facing Files list
- Skills can work with a chat-uploaded PDF as a normal file
- Knowledge remains a separate product plane
- object-storage paths and raw sandbox paths are not normal product/model selectors

## Consequences

### Positive

- users get one mental model: Files
- the assistant gets one stable working-file alias surface, resolved by runtime to canonical `fileRef`
- generated outputs become reusable after creation without turn-local special cases
- file search can honestly mean "search user-visible assistant files"
- storage backend details stop leaking into prompts and tool arguments
- later Files UI can be built on one registry instead of reconciling chat media, sandbox, and runtime artifacts client-side

### Negative

- existing partial `artifactId` and lazy attachment-registration paths must be removed, not wrapped
- media delivery and generated-artifact persistence need to create canonical file rows earlier
- chat attachment schemas likely need a durable link to `AssistantFile`
- tests across API/runtime/web will change because `attachmentId` and `artifactId` stop being model-facing handles

## Alternatives considered

### Keep current split and improve instructions

Rejected. The current split leaks too many scopes into the model: `attachmentId`, `artifactId`, `fileRef`, `objectKey`, and relative path. Better prompt wording cannot make an inconsistent selector model reliable.

### Add a resolver while preserving all selector types

Rejected as target architecture. A resolver can be useful internally, but if the model contract continues to accept every historical id, the assistant still has to choose among scopes. That preserves the confusion instead of removing it.

### Treat Knowledge documents as Files

Rejected. Knowledge documents are indexed retrieval sources with processing status, chunking, quality, and governance. Files are visible/reusable artifacts. They may link internally, but they are not the same product surface.

### Folder-first user model

Rejected as primary UX. Folders imply storage implementation and hierarchy as the main concept. Provenance filters over one Files list better match the product goal: the user works with files, not with backend storage systems.

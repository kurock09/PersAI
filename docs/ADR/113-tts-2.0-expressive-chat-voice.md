# ADR-113: TTS 2.0 — expressive, controllable chat voice replies

## Status

Complete. Baseline SHA: `e124c8575e52515f7e989bab557e46ff9af4abe0` (clean tree at session start).

This is a normal implementation ADR (not an orchestration program): the active agent writes production code directly, one bounded slice per session.

Delivered in Slices 1, 2, and 3a. Slice 3b (`eleven_v3` test-phrase synthesis) was **dropped** — see the slice plan note below. The original TTS 2.0 scope (expressive `eleven_v3` path, structured intent + safe compiler, provider-gateway quality path, voice catalog cache, premium picker) is closed.

## Context

Chat voice/audio replies already work through the native `tts` worker tool: runtime executes the tool, provider-gateway calls the speech provider, the artifact is persisted as an `AssistantFile` audio output, and web/Telegram deliver it through the existing media/job flow. ADR-110 Slice 3 made TTS model truth catalog-driven for OpenAI and kept the runtime provider-chain fallback (`elevenlabs -> yandex -> openai` and friends).

The current path has three product-quality gaps for "living" voice:

1. **Delivery is flat.** The model-facing `tts` tool only exposes a single `toneTag` enum (`neutral|warm|gentle|calm|cheerful|playful|confident`). There is no structured way to express emotion, pace, intensity, pauses, or non-verbal sounds, so voice replies sound uniform.
2. **ElevenLabs runs on `eleven_multilingual_v2`.** The expressive `eleven_v3` model (inline audio tags such as `[whispers]`, `[excited]`, `[laughs]`) is not used, and there is no safe path to drive those tags. Letting the model write raw ElevenLabs tags directly would be unsafe (prompt-junk / unsupported-tag risk).
3. **Voice selection UX is a plain backend list + select.** `ResolveAssistantVoiceSettingsService` already returns normalized ElevenLabs voices (gender/category/preview URL/load-error state) live per request, but there is no cache/shortlist service and the web picker is a basic select rather than a premium card picker with preview, search/filter, and test-phrase generation.

## Decision

Build TTS 2.0 as a PersAI-native, PROD-ready upgrade of the existing chat voice path. Keep the `tts` worker tool, the media/job delivery flow, and the provider fallback chain. Replace flat tone steering with a **structured expressive delivery intent** that a **safe compiler** turns into conservative ElevenLabs `eleven_v3` audio tags, and upgrade the ElevenLabs voice catalog/picker to a premium v2 experience.

### Structured TTS intent (model-facing, not raw tags)

The `tts` tool accepts these structured, enum-constrained fields (all optional, each defaulting to a neutral value):

- `delivery`: `neutral | calm | warm | confident | playful | dramatic | whisper | narrator`
- `emotion`: `neutral | happy | sad | excited | serious | curious`
- `pace`: `slow | normal | fast`
- `intensity`: `low | medium | high`
- `pause`: `none | short | long`
- `nonVerbal`: `none | laugh | chuckle | sigh | clear_throat`

The model never writes raw ElevenLabs tags. It only chooses structured intent; the runtime/provider-gateway own the translation.

### Safe compiler (structured intent -> conservative Eleven v3 tags)

A pure, deterministic compiler converts structured intent into a small, conservative set of `eleven_v3` audio tags:

- `delivery=whisper` -> `[whispers]`
- `delivery=dramatic` -> `[dramatic]` (only when not suppressed by a conflict)
- `emotion=excited` -> `[excited]`
- `emotion=sad` -> `[sad]`
- `emotion=curious` -> `[curious]`
- `nonVerbal`: `laugh -> [laughs]`, `chuckle -> [chuckles]`, `sigh -> [sighs]`, `clear_throat -> [clears throat]`
- `pause`: `short -> [short pause]`, `long -> [long pause]`

Guardrails:

- **Conflict avoidance.** `whisper` suppresses `[excited]` and `[dramatic]`; a `high` intensity escalation is ignored under `whisper` (whisper stays calm and quiet).
- **Tag budget.** At most `MAX_ELEVEN_V3_TAGS = 3` tags per generation, selected by a fixed priority (delivery > emotion > nonVerbal > pause) so output stays natural.
- **No model-authored tags.** On the v3 path, bracketed `[...]` tag-like sequences are stripped from the spoken `text` before the compiled tags are prepended, so the model cannot inject arbitrary tags through the text field.
- **Provider safety.** For non-ElevenLabs providers and for ElevenLabs models that are not `eleven_v3`, the v3 tags are ignored and the path falls back to the existing tone-based `voice_settings` behavior.

`pace` and `intensity` do not map to v3 tags in the safe set (no stable v3 tag exists for them). They influence the derived legacy tone and conservative v3 `voice_settings` (`stability`) instead.

### Provider-gateway eleven_v3 quality path

ElevenLabs is the primary quality provider. The provider-gateway resolves the model from the catalog-owned `credential.modelKey` when present (ADR-110), otherwise defaults to `eleven_v3`. On the v3 path it prepends the compiled tags, sanitizes model-authored tags from text, and sends conservative discrete-stability `voice_settings`. On non-v3 ElevenLabs models it keeps the prior `eleven_multilingual_v2`-style full `voice_settings` behavior. The saved ElevenLabs `voiceId` from the assistant voice profile remains the voice source.

### Backward compatibility

- The internal `ProviderGatewaySpeechGenerateRequest` keeps a required `toneTag` (now **derived** by the runtime from the structured intent) so Yandex / OpenAI tone baselines keep working unchanged.
- The persisted assistant voice profile schema (`persai.assistantVoiceProfile.v1`) is unchanged; the saved ElevenLabs `voiceId` is reused.

### Voice catalog v2 + premium picker (later slices)

- Backend catalog/cache service with normalized metadata, load/error state, preview URL, and a setup/settings shortlist.
- Web premium picker: voice cards, play preview, search/filter (gender/category/language), `eleven_v3` test-phrase generation, and save into the assistant voice profile.

## Scope

In:

- `eleven_v3` as the primary quality model for chat TTS.
- Structured expressive delivery intent on the `tts` tool + safe compiler.
- ElevenLabs voice catalog v2 backend/cache + premium picker UI (later slices).
- Web chat + Telegram delivery continue through the existing media/job flow.

Out:

- HeyGen TTS replacement, HeyGen avatar/talking-video changes, HeyGen Studio API.
- ElevenLabs Sound Effects API, background music/noise, audio mixing.
- Programmatic audio quality scoring.
- Broad refactors unrelated to chat TTS.

## Slice plan

- **Slice 1 (done) — expressive delivery core.** Structured TTS v2 intent in `@persai/runtime-contract`; safe `eleven_v3` tag compiler in provider-gateway; ElevenLabs client `eleven_v3` model + compiled-tag prepend + text sanitize + conservative v3 voice settings; runtime `tts` tool accepts/passes structured intent and derives the legacy `toneTag`; tool projection exposes structured fields; tests across contract/runtime/provider-gateway; docs.
- **Slice 2 (done) — voice catalog v2 backend/cache.** Platform-wide `ElevenLabsVoiceCatalogService` backed by `platform_elevenlabs_voice_catalog_cache` (24h TTL, lazy refresh, stale-on-failure fallback); normalized metadata (gender, category, language + `ru`/`en`/`other` bucket, preview URL); honest `ready`/`not_configured`/`unavailable` load state. `GET assistant/voice/settings` reads the cache instead of a live per-request fetch and additively exposes per-entry `language`/`languageBucket`. The active refresh source is the ElevenLabs shared voice library (`/v1/shared-voices`), bounded into a small curated set per language bucket and gender with a 50/50 mix of featured and popularity-ranked candidates. (An earlier draft also exposed a `shortlist`/`fetchedAt`; both were dropped in the same-day cleanup audit as unconsumed.)
- **Slice 3a (done) — premium voice picker UI.** Shared `VoicePicker` list component (`apps/web/app/app/_components/voice-picker.tsx`) unified across all three TTS providers in assistant settings. ElevenLabs uses a strict `RU | EN | OTHER` bucket switch plus a compact name + preview-play list; assistant gender remains the upstream constraint from assistant settings rather than another picker control. Yandex/OpenAI use the same compact list without language buckets. Honest `loading`/`not_configured`/`unavailable` states. Save still writes the per-provider voice into `persai.assistantVoiceProfile.v1`.
- **Slice 3b — `eleven_v3` test-phrase generation. DROPPED (2026-06-08).** Was planned as an authenticated, quota-metered on-the-fly synthesis of a sample phrase. Dropped as unnecessary product complexity: choosing a voice does not justify a paid synthesis endpoint plus billing/quota plumbing and anti-abuse throttling. The Slice 3a card stock-preview (`previewUrl`) already answers "how does this voice sound", and real intonation/expressivity is verified live — the assistant sends an actual voice note in chat, which is the authoritative signal. No metered preview seam is added.

## Consequences

- ElevenLabs chat TTS defaults to `eleven_v3` (intentional replacement of the `eleven_multilingual_v2` default) unless an explicit catalog `modelKey` overrides it.
- Voice quality/expressivity is owned by structured intent + a conservative compiler, not by free-form model text, which keeps output natural and safe.
- Yandex/OpenAI TTS behavior is preserved via a derived legacy tone tag.

## Founder directive

Every chat file must have a short semantic description so the model understands file **content**, not only filename and path, and can resolve natural-language references such as *"find my photo in a cap"* or *"the excel with competitors"* to an exact workspace path for `files.attach`, `image_edit`, and `document.*`.

This is the **semantic index** program. Working Files is one read surface; the durable product truth is `workspace_file_metadata.shortDescription`.

Requirements:

- one pipeline for all file kinds (image, pdf, xlsx, py, cpp, audio, ‚Ä¶) ‚Äî no product-level type split;
- one source of truth ‚Äî no mirror in `attachment.metadata`;
- path identity only ‚Äî no `AssistantFile` registry or `assistantFileId` jobs;
- clean PROD cutover ‚Äî no legacy upload-micro-description services, no dual write paths.

## Decision

### D1 ‚Äî Single source of truth

The only durable semantic index field is:

```text
workspace_file_metadata (workspaceId, path).shortDescription
```

Provenance may be recorded in job/upsert internals, but **no** duplicate in `assistant_chat_message_attachment.metadata`, no revived `AssistantFile` row, no second cache the model or UI must reconcile.

### D2 ‚Äî One write pipeline, three layers

When a file appears in workspace object storage and has a manifest row:

1. **Deterministic (sync, no LLM)** ‚Äî if STT transcription or text_extract yields a usable one-liner ‚Üí upsert `shortDescription` immediately; do not enqueue LLM.
2. **Generation request (sync, runtime/API)** ‚Äî for model-produced files (`files.write`, image/document delivery, etc.): `buildGeneratedFileSemanticSummary({ requestText, requestedName, ... })` ‚Üí if informative ‚Üí upsert with source `generation_request`; if weak/empty ‚Üí fall through to layer 3.
3. **Background cheap-LLM job (async)** ‚Äî enqueue by `(workspaceId, path)` when summary is still empty and policy allows (see D3). Worker downloads bytes from object storage, runs `WorkspaceFileMicroDescriptionService`, upsert manifest. User turns must not block on job completion.

All file kinds use this same pipeline. Implementation may choose different byte-to-prompt strategies internally (text preview, image block, pdf block, lightweight office extract); that is not a product-facing type split.

### D3 ‚Äî Enqueue policy

| Source | Enqueue when summary empty |
|--------|---------------------------|
| Project-mode upload / inbound | **Always** |
| Ordinary / B2C upload / inbound | Only if `routerPolicy.analyzeUploadsOnB2cUpload === true` |
| Model-generated (`files.write`, delivery, attach binary) | **Always** (no B2C gate) |

Dedup rules:

- Non-empty `shortDescription` ‚Üí never enqueue.
- Completed job with null summary (unsupported type, timeout, max attempts) ‚Üí do not retry forever.

### D4 ‚Äî Background job model (path-keyed, new table)

Do **not** restore `assistant_upload_micro_description_jobs` or `assistantFileId`.

New leased-job table `workspace_file_micro_description_jobs` keyed by `@@unique([workspaceId, path])` with standard scheduler claim fields (`status`, `attempt`, `schedulerClaim*`, `usageJson`, `usageOccurredAt`, ‚Ä¶). Pattern matches existing API schedulers (leased tick ‚Üí claim ‚Üí process ‚Üí complete/fail).

Helper service `WorkspaceFileMicroDescriptionService`:

- system-tool slot from runtime bundle (same class of helper as pre-v3 upload analysis);
- bounded: ~20s timeout, ~60 output tokens, ~4000 char text preview, ~4MB binary cap;
- strict JSON output `{ summary: string }`; empty string when cannot describe;
- cost ledger: `purpose=tool_helper`, `source=upload_micro_description`, non-blocking.

### D5 ‚Äî Enqueue points (mandatory)

After object bytes exist and manifest row is upserted, wire enqueue (or sync layer 1/2) at:

- web upload / stage ‚Äî `manage-chat-media.service.ts`
- inbound telegram / channel media ‚Äî `inbound-media.service.ts`, `handle-internal-telegram-turn.service.ts`
- runtime metadata upsert ‚Äî `upsert-workspace-file-metadata-from-runtime.service.ts` (`files.write`)
- outbound delivery ‚Äî `media-delivery.service.ts`
- explicit attach registration ‚Äî `register-chat-attachment.service.ts`

No shadow enqueue path. No attachment-metadata side channel.

### D6 ‚Äî Read surfaces (one field, three windows)

`shortDescription` is joined at read time into:

1. **`files.list`** ‚Äî already wired; verify end-to-end after write pipeline lands.
2. **Working Files developer block** ‚Äî batch-join manifest descriptions for **all** current-session file paths (attachments + discovered paths), set `semanticSummaryHint` and honest `createdAt` / author on each line. Fix lives in `turn-context-hydration.service.ts`.
3. **`files.search`** ‚Äî restore as seventh `files` action. Search `workspace_file_metadata` by tokenized match on `path`, display name (basename), and `shortDescription`; rank in memory; return items with `shortDescription` and discovered aliases (`found file #N`). API: `POST /api/v1/internal/runtime/files/search`. Update runtime tool, contract enum, native projection, tool catalog, and Working Files recovery text in `turn-execution.service.ts` (*If user refers to a file not in Working Files ‚Üí `files.list` ‚Üí `files.search` ‚Üí only then unavailable*).

### D7 ‚Äî Instruction ownership

Active model-facing surfaces must agree:

- semantic hints come from manifest `shortDescription` only;
- `files.search` is a real action again;
- no guidance referencing `fileRef`, `AssistantFile`, or pre-v3 registry search.

## Acceptance criteria

Closure requires all of the following on a clean tree after S1‚ÄìS7:

1. **Upload + policy:** B2C chat, toggle ON, PNG upload ‚Üí `shortDescription` populated in manifest within ~30‚Äì60s.
2. **Upload + policy:** B2C chat, toggle OFF ‚Üí no LLM job enqueued (deterministic-only if applicable).
3. **Upload + policy:** Project-mode upload ‚Üí LLM job always enqueued when summary empty.
4. **Discovery:** Query *"find my photo in a cap"* via `files.search` returns the correct path when manifest summary matches.
5. **List:** `files.list` items include `shortDescription` when manifest has it.
6. **Working Files:** Session attachment lines show non-empty microDescription and honest createdAt (not `unknown | - | -`).
7. **Generated:** `files.write` xlsx with strong request text ‚Üí `shortDescription` via generation_request and/or background job.
8. **Generated:** Image delivery output gets summary (generation_request or job).
9. **Code/text:** Upload `.py` / `.cpp` uses same pipeline; summary describes content.
10. **Non-regression:** `analyzeUploadsOnB2cUpload` ‚âÝ Completion vision; no attachment.metadata mirror; paths follow ADR-133 hierarchy.
11. **Cleanliness:** Zero active code reviving `AssistantFile` micro-description jobs or dual semantic-index writes.

## Risks (accepted)

- Office/xlsx extract is best-effort; null summary after failed job is honest.
- Job lag 30‚Äì60s ‚Äî WF/list refresh on next turn is acceptable.
- Search with empty summaries returns honest empty results; model falls back to list/read per D6 recovery text.

## Addendum (2026-07-04) ‚Äî WF cleanup + summary invalidation + version markers

**Status:** implemented locally on top of S1‚ÄìS7.

### D8 ‚Äî Invalidate stale summaries on content change

When manifest content changes at the same path, clear `shortDescription` and re-enqueue the background job with `forceRefresh`:

- runtime manifest upsert: `replace: true` or `contentHash` delta;
- register/upload attach path: `sizeBytes` or `mimeType` delta when no new deterministic summary is supplied.

`enqueueIfNeeded({ forceRefresh: true })` bypasses the `summary_exists` short-circuit.

### D9 ‚Äî Working Files presentation cleanup

Remove legacy developer blocks (`DOC_CURRENT_SOURCE`, `DOC_LAST_DELIVERED_PDF`, `LAST_DELIVERED_FILE = ...`). Express roles only as row markers:

- `last delivered` ‚Äî newest user/model file in chat;
- `current source` ‚Äî newest user document-source upload (pdf/docx/xlsx/text/*);
- `v{N}` ‚Äî current registered document version for pdf/docx/xlsx paths.

Pinned rows with `last delivered`, `current source`, or `v{N}` survive Working Files trim.

### D10 ‚Äî Version hint join

`POST /api/v1/internal/runtime/files/short-descriptions` also returns `documentVersionNumber` for registered pdf/docx/xlsx paths. Working Files and discovered-path hydration consume it; `files.list` / `files.search` remain path+summary only.

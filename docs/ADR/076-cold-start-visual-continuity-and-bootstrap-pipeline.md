## Context

The current `apps/web` app shell has three perceptible cold-start artifacts that the founder explicitly called out (web and mobile both):

1. **Theme flash** ‚Äî on a cold load the WebView paints once with the CSS `:root` defaults (dark) before the inline `<head>` bootstrap script can apply the `light` class. On Capacitor specifically, the native WebView background defaults to white, so a user in dark mode sees a white frame before HTML, then a brief dark-default frame, and only then the resolved theme.
2. **Empty shell ‚Üí data pop-in** ‚Äî `apps/web/app/app/_components/app-shell.tsx` is `"use client"` and `useAppData` (`apps/web/app/app/_components/use-app-data.ts`) fires six parallel REST calls (`getAssistant`, `getAssistantWebChats`, `getAssistantTelegramIntegration`, `getAssistantNotificationPreference`, `getAssistantPlanVisibility`, `getAdminPlanVisibility`) only after the client mounts. Until those resolve the sidebar / account footer / chat header are empty.
3. **Avatar pop-in** ‚Äî `apps/web/app/app/_components/assistant-avatar.tsx` fetches the assistant avatar in JS as a Bearer-authenticated blob (`getToken()` ‚Üí `fetch(url, { Authorization })` ‚Üí `res.blob()` ‚Üí `URL.createObjectURL`). The cache (`AVATAR_BLOB_CACHE`) is in-memory only, so every full reload ‚Äî and therefore every Capacitor cold-start ‚Äî re-issues the round-trip before the avatar appears. The `apps/api` route already returns `Cache-Control: no-cache, must-revalidate`, so the browser HTTP cache is intentionally disabled on it.

Adjacent symptoms in the same wave:

- `<meta name="theme-color">` is keyed off `prefers-color-scheme`, not the in-app theme choice ‚Äî the OS bar can be one tone different from the app frame for the rest of the session if the user's explicit choice diverges from OS preference.
- Slide-overs (`AssistantSettings`, `TelegramConnect`) are mounted in `AppShell` regardless of open state, paying their full client bundle cost on first paint.
- Android `styles.xml` references `@drawable/splash` that does not exist in the resource set, so the launch theme falls back to the platform default white.

The constraint is the same as ADR-075's: **the WebView shell loads `https://persai.dev` directly**. There is no separate "mobile build" we can optimize differently. Anything we ship to web reaches mobile on next app open and vice versa, so the architecture must be uniformly correct on both.

The product policy stated by the founder is explicit: no "fast fixes", no transitional or dual-mode patches, no legacy tail. Each problem area is solved by the architecturally correct shape, with the previous shape removed in the same slice.

## Decision

Adopt a coherent six-slice program (with a conditional seventh) that replaces the current cold-start pipeline with one where the rendered HTML is already correct on the first paint and where avatars + bootstrap data arrive through HTTP-cacheable, server-prefetched paths.

The slices are tightly coupled by intent but loosely coupled by code, so each lands as one bounded session and one commit per repo, with the verification gate green and all docs updated in the same slice.

### Slice 1 ‚Äî Cookie-as-truth theme with SSR-baked `<html class>`

The user's theme choice (`system | light | dark`) becomes a cookie (`persai-theme`, `Max-Age=31536000`, `Path=/`, `SameSite=Lax`, `Secure` on production), with `localStorage` kept as a same-origin mirror written from the controller. The Server Component `apps/web/app/layout.tsx` reads the cookie at request time and writes the resolved class onto `<html>` directly, plus the resolved `<meta name="theme-color">` value matching the active palette `--chrome` token.

Cookie absence (first visit, brand new install) is handled in two layers:

- on Capacitor Android (Chromium WebView) and modern Chrome/Edge, `Sec-CH-Prefers-Color-Scheme` client hint is opted into via `<meta http-equiv="Accept-CH" content="Sec-CH-Prefers-Color-Scheme">` plus a `Critical-CH` response header on the root document, so the second navigation already carries the OS preference and the server resolves correctly without any JS;
- on iOS WKWebView and any older browser that does not send the client hint, a tiny inline script in `<head>` performs the same single-frame resolution and writes the cookie so subsequent navigations stay server-resolved. This is the only piece of inline JS retained, and it is the documented first-visit-without-cookie fallback, not a parallel source of truth.

`apps/web/app/app/_components/use-theme.ts` keeps owning runtime updates (toggle, OS-change subscription) but writes both cookie and `localStorage` on every change. The cookie is the authoritative source on the next request; `localStorage` is the synchronous client-side read for the same request lifecycle.

Result: dark/light/system flicker on cold start is structurally impossible because the HTML arrives with the right class already on `<html>`.

### Slice 2 ‚Äî Capacitor cold-start visual continuity (mobile)

Slice 2 has two halves, both shipping in `persai-mobile`:

**2a ‚Äî runtime system-bars bridge (LANDED).** The Android system status bar and bottom navigation bar are OS chrome and are not influenced by `<meta name="theme-color">`. A `SystemBarsBridge` JavaScript interface (`window.PersaiNative.setTheme()`) is registered on the WebView in `MainActivity#onCreate`; `apps/web/app/app/_components/use-theme.ts` calls it from inside `applyResolved()` so every theme transition atomically updates both the web surface and the native bars. Cold-start pre-coloring is driven by a `SharedPreferences` write path so the bars are correct before the first WebView frame, even when the user's in-app choice diverges from OS preference. Full design is documented in ADR-075 ("System bars theme bridge"); this avoids re-statement here.

**2b ‚Äî splash and WebView background (PROPOSED).** `persai-mobile/capacitor.config.ts` declares `android.backgroundColor` and `ios.backgroundColor` matching `--chrome` dark (`#161513`). The Android launch theme stops referencing the missing `@drawable/splash` and uses a real PersAI launch drawable on `#161513`; if a light variant is desired, it ships under `values-night/styles.xml` with the inverted `--chrome` light token (`#e0d8c8`). iOS gets the equivalent via `LaunchScreen.storyboard` (PersAI wordmark on `#161513`).

Together 2a and 2b eliminate every visual discontinuity from native splash to the first HTML paint. Slice 1 makes sure that what lands after the splash is already in the correct theme; 2a makes sure the system bars match it from frame zero; 2b removes the existing `@drawable/splash` placeholder and the white WebView frame.

### Slice 3 ‚Äî RSC bootstrap + single batch endpoint

`apps/api` adds a new bounded endpoint:

- `GET /api/v1/app/bootstrap` ‚Äî Bearer-protected (existing `ClerkAuthMiddleware`), fans out internally to the same six application services that today are six separate routes (`getAssistant`, `getAssistantWebChats`, `getAssistantTelegramIntegration`, `getAssistantNotificationPreference`, `getAssistantPlanVisibility`, plus the admin probe), runs them in parallel via `Promise.allSettled`, and returns one envelope `{ requestId, sections: { assistant, chats, telegram, notificationPreference, plan, admin } }` where each section is `{ ok: true, data } | { ok: false, error }`. Partial failures are explicit per section instead of collapsing the whole bootstrap.

`apps/web/app/app/layout.tsx` becomes an `async` Server Component that:

- reads the Clerk session from cookies via `auth()` and obtains a server-side token via `auth().getToken()`,
- calls `bootstrap` once,
- passes the resolved sections to `<AppShell initialData={...}>`.

`useAppData` is refactored to accept `initialData` and start in the resolved state (`isLoading: false`, all sections populated) instead of `isLoading: true`. It still owns `reload()` / `reloadChats()` for explicit refreshes after mutations, but there is no client-side initial fan-out anymore. The six per-endpoint client wrappers in `assistant-api-client.ts` stay alive only as inputs to refresh paths; if any of them ends up unused, it is deleted in the same slice (no dead exports).

Failure modes:

- if `bootstrap` itself returns 401 (Clerk session expired during SSR), `apps/web` renders the existing Clerk redirect path ‚Äî same as today.
- if individual sections return `{ ok: false }`, the relevant UI surface renders its existing error/empty state ‚Äî same envelope shape that the per-endpoint client paths already produce.

Result: no client-side "isLoading ‚Üí loaded" transition on cold start; sidebar / account footer / chat header are populated in the first paint, on web and mobile.

### Slice 4 ‚Äî Cookie-auth versioned avatar pipeline

The avatar pipeline becomes a same-origin, HTTP-cacheable static asset:

- `apps/web` adds a Next.js route handler at `apps/web/app/api/avatar/[hash]/route.ts` that authenticates via the Clerk cookie session (`auth()` from `@clerk/nextjs/server`), fetches the avatar bytes server-side from `apps/api` (using the server-side bearer token), and streams the response back with `Content-Type` from upstream and `Cache-Control: private, max-age=31536000, immutable`, `ETag: "<hash>"`. Because the URL is content-addressed by `hash`, cache-busting on avatar change is automatic.
- `apps/api` lifecycle state changes: `assistant.draft.avatarUrl` and `assistant.published.avatarUrl` move from `https://<api-host>/api/v1/assistant/avatar` to a stable, content-addressed URL `/api/avatar/<hash>.<ext>` (relative to the web origin). The hash is derived deterministically from the stored avatar content (e.g. SHA-256 prefix) and is part of the lifecycle state envelope. The previous bearer-protected `GET /api/v1/assistant/avatar` route is removed in the same slice; the upload endpoint (`POST /api/v1/assistant/avatar`) stays and continues to write the bytes plus recompute the hash.
- `apps/web/app/app/_components/assistant-avatar.tsx` collapses to a thin presentational component: explicit `width`/`height` (per `size`), a fixed-size container with the existing `bg-accent/15` fallback that contains either the emoji or the `Sparkles` icon, and `<img src={avatarUrl} loading="eager" decoding="async" />` when `avatarUrl` is non-null. The `AVATAR_BLOB_CACHE`, the `useAuth().getToken()` flow, the `fetch + blob + URL.createObjectURL` pipeline, and the `useEffect` that drives them are all removed.

Result: avatars hit the browser HTTP cache on cold start (after the first ever load), there is no JS-driven fetch, and there is one canonical place that serves the bytes (the BFF route in `apps/web`).

### Slice 5 ‚Äî Skeleton contracts only where data is genuinely pending

After Slice 3, the only genuinely pending states are mid-session refreshes (e.g. `reloadChats()` after a chat is created or deleted, or thread switching). These are short and bounded.

This slice introduces small, targeted skeleton placeholders only where these pending states actually surface ‚Äî for example, a 2‚Äì3 line shimmer in the sidebar chat list while `reloadChats()` is in-flight after a destructive action. There is no global "isLoading" skeleton wave because there is no global loading state on cold start. If a surface has no genuine pending window, it gets no skeleton.

### Slice 6 ‚Äî Slide-over code-splitting and bundle audit

`apps/web/app/app/_components/app-shell.tsx` mounts `AssistantSettings` and `TelegramConnect` even when their slide-overs are closed. Both become `next/dynamic({ ssr: false, loading: () => null })` imports, mounted only when `settingsOpen` / `telegramOpen` is true. Same treatment for any sibling component (lightbox, integration-specific panels) that is mounted-but-hidden in the initial route bundle.

In the same pass, an explicit one-shot audit:

- lucide-react imports stay named (already true, but verify the chunk graph has not regressed),
- decide per-component whether `framer-motion` carries weight ‚Äî if a use site can be replaced by a CSS transition without degrading UX feel, replace it; otherwise keep `framer-motion` and document why.

Outcome target: measurable initial-route JS reduction, no regression in interaction feel.

### Slice 7 ‚Äî Service-Worker PWA shell (conditional)

After Slices 1‚Äì6 land and we have a measured baseline, we evaluate whether second-cold-start latency on Capacitor still justifies a Service Worker shell. The decision criteria ‚Äî and the platform support matrix (Android Chromium WebView: full SW; iOS WKWebView: limited / version-gated) ‚Äî are deliberately deferred to the post-Slice-6 evaluation rather than committed here. If we proceed, the slice is bounded to `apps/web` (no `persai-mobile` change) using `next-pwa` or an equivalent Workbox-based recipe scoped to static assets and the app shell. If we do not proceed, this entry stays in the ledger as `decided not to ship`.

## Consequences

### Positive

- **No theme flicker.** SSR-baked `<html class>` plus theme-color meta are correct in the first byte of HTML; the WebView's first paint is already in the right palette.
- **No empty-shell pop-in on cold start.** Bootstrap data is fetched server-side and rendered into the first paint; the client never enters `isLoading=true` for the initial mount.
- **Avatars are HTTP-cached.** Versioned, immutable URLs let the browser do the work that the JS blob cache used to do; cold reloads no longer re-issue the round-trip.
- **Single canonical avatar path.** One BFF route in `apps/web`, one upload route in `apps/api`. No two-path drift.
- **Single bootstrap surface.** One endpoint replaces six client-fired round-trips at startup; mutations still use the per-endpoint refresh paths, so the per-endpoint contracts stay intact.
- **Smaller initial JS.** Slide-overs no longer pay their bundle cost when closed.
- **Mobile parity for free.** Slices 1, 3, 4, 5, 6 land in `apps/web` and reach `persai-mobile` on the next app open. Only Slice 2 touches the mobile shell.

### Negative

- **More server work per cold start.** `bootstrap` runs six application services on the API, and `apps/web` now does an extra server-side fetch in `app/app/layout.tsx`. This is bounded ‚Äî the same six calls used to run on the client, so total work is roughly conserved ‚Äî but the API-side latency budget for cold start is now tighter than before. We mitigate by making bootstrap idempotent and `Promise.allSettled`-fanned, so a slow non-critical section does not hold up the rest.
- **Cookie-bound theme adds a request-level coupling.** Reading `persai-theme` cookie in every request is cheap, but it becomes part of the cache vary surface for any cached SSR. PersAI does not currently SSR-cache authenticated app routes, so the impact is zero today; if SSR caching is ever introduced for the app shell, the theme cookie joins the cache key.
- **Avatar URL format changes inside lifecycle state.** Any external consumer (admin tooling, embeds) reading `assistant.draft.avatarUrl` directly will see the format flip from `https://<api-host>/api/v1/assistant/avatar` to `/api/avatar/<hash>.<ext>`. There are no documented external consumers today; if any exist they would have been failing already because the old URL was Bearer-protected and not browser-shareable.
- **One more Next.js route handler in the web BFF.** Trivial maintenance cost; keeps the boundary clean.

### Out of scope / non-goals

ADR-076 does not:

- introduce SSR caching for authenticated app routes,
- change Clerk's auth model (proxy mode stays on `/clerk-proxy`, Bearer stays on `apps/api`),
- change the multichannel runtime contract (telegram/web bootstrap reads from the same lifecycle services),
- introduce any "transitional" mode that supports both the old and new avatar URL shape simultaneously,
- introduce any "transitional" mode that supports both the old client fan-out and the new bootstrap simultaneously,
- alter the offline behaviour (`<OfflineGate />`, `offline.html`) defined in ADR-075,
- alter the single-slot pending send behaviour defined in ADR-075.

## Alternatives considered

- **Quick fixes only (`localStorage` early-read, in-memory avatar cache, sidebar skeletons).** Rejected by founder ask. Each of these would leave a parallel "old" path alongside the corrected one, which is exactly the residue this ADR is meant to remove.
- **Service-worker shell as the primary fix.** Rejected as the first move. SW would mask the symptoms (cold-start payload) without fixing the architectural shape (client-fired bootstrap, JS-driven avatar fetch, theme as JS-resolved property). Slice 7 keeps the option open as a layered improvement once the underlying shape is clean.
- **`next-themes` for the theme layer.** Rejected. Brings a dependency for a contract we already implement in 90 lines (`use-theme.ts`); the cookie-as-truth design is a small explicit change to the existing controller and does not justify an external library on this slice.
- **Avatar served from `apps/api` with cookie auth.** Rejected. The API surface is uniformly Bearer-only by design (single auth pattern, single audit boundary). Moving one route to cookie auth would create a second auth-shape inside the API. The chosen design keeps API auth uniform and puts the cookie‚Üíbearer translation in the web BFF where same-origin cookie reading is the canonical pattern.
- **Single `bootstrap` endpoint that also embeds the avatar bytes inline (data-URL).** Rejected. Avatars belong in the HTTP cache; embedding them in the bootstrap envelope makes every bootstrap response uncacheable and bloats the SSR payload.

### Combined-slice batches (founder-authorised 2026-04-25)

To avoid multiplying founder live-verification gates without trading away rollback safety, slices that share their delivery surface and whose scope does not cross the mobile-binary boundary land in single batches:

- **Batch A ‚Äî Slice 3 + Slice 4 (single session, single deploy, single founder gate).** Both slices change `packages/contracts/openapi.yaml`, share one orval-regenerate cycle, and the new lifecycle envelope from Slice 4 (`assistant.{draft,published}.avatarUrl` ‚Üí `/api/avatar/<hash>.<ext>`) rides through the `bootstrap` envelope from Slice 3 ‚Äî they are logically coupled and uncoupling them across two sessions duplicates the contract-regeneration round trip without any safety win. The combined founder live gate covers both targets: (a) all six sections present in first paint with no `isLoading=true ‚Üí loaded` on cold start, (b) cold reload reuses the HTTP cache for the avatar (no JS round-trip) and a fresh upload produces a new content hash ‚Üí new URL ‚Üí fresh fetch.
- **Batch B ‚Äî Slice 5 + Slice 6 (single session, single deploy, single founder gate).** Both slices are `apps/web`-only presentational / bundle work and depend on the same Slice 3 baseline. The combined founder live gate covers: (a) zero skeletons on cold start, brief sidebar shimmer only after destructive actions, no other surface enters a global loading state; (b) measurable initial-route JS reduction (bytes-before / bytes-after recorded in CHANGELOG) and unchanged interaction feel on the slide-overs, footer popup, and sidebar overlay.
- **Standalone ‚Äî Slice 2 (residual 2b)**. Mobile binary; `cap sync` + APK rebuild + Z Fold 6 live test. Cannot batch with web-only work.
- **Standalone ‚Äî Slice 7 (conditional)**. Decision happens after Batch B lands and a measured Slice 6 baseline is captured.

Batches A and B do not relax any individual slice's acceptance criteria; they only collapse the founder-gate cadence from per-slice to per-batch. If a problem surfaces on `persai-dev` after a batch lands, both slices in the batch revert together ‚Äî that trade-off is deliberate and was accepted alongside the batching authorisation.

## Section 3 ‚Äî Independence proof for `Enabled Skills` prompt block

The materialized `Enabled Skills` block (`apps/api/src/modules/workspace-management/application/enabled-skills-prompt-materialization.ts`) is the only piece of code that decides whether a Skill's `instructionCard` is visible to the model on a given turn. Its decision is **purely** assignment-state based ‚Äî it never reads runtime `skillDecisionState`, `activeSkillId`, or any cadence row.

The filter is on `assignmentStatus === "active"`, **not** runtime decision state:

```52:56:apps/api/src/modules/workspace-management/application/enabled-skills-prompt-materialization.ts
  const enabled = params.candidates
    .filter((candidate) => candidate.status === "active")
    .filter((candidate) => candidate.assignmentStatus === "active")
    .sort(compareEnabledSkillCandidates);
```

`assignmentStatus` is sourced from `AssistantSkillAssignment.assignmentStatus` (per ADR-079); it flips only when the admin/user toggles a Skill on or off in Settings. The candidate row shape (`EnabledSkillPromptCandidate`, `apps/api/src/modules/workspace-management/application/enabled-skills-prompt-materialization.ts:14-26`) does not even carry a runtime decision-state field.

Therefore:

- **Enabling a Skill in Settings already makes its `instructionCard` visible to the model on every turn**, regardless of whether `skillDecisionState.status === "active"` and regardless of any cadence row.
- ADR-118 only changes the **activation effect** ‚Äî retrieval-stage priority (`orchestrate-runtime-retrieval`), Skill-ref cache key (`SkillRetrievalStateService.clearForChatWhenSkillMismatches`), and the new volatile scenario block ‚Äî **not** instruction visibility.

Conclusion: **the cached `Enabled Skills` prompt block is independent of `skillDecisionState`**; ADR-118 can replace the activation writer (cadence/classifier ‚Üí `skill` tool) without changing which Skill cards the model sees in the cached prefix.

## Section 4 ‚Äî Volatile-context end-to-end trace

**1. Definition site (contract).** The flag is declared as an optional field on `ProviderGatewayTextMessage`:


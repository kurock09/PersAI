## 2026-06-06

- History compaction: `docs/CHANGELOG.md` now keeps only current entries and concise recent summaries. Detailed 2026-06-05-and-earlier entries moved to `docs/CHANGELOG.archive-2026-06-05-details-and-earlier.md`; older pre-existing archive remains unchanged.
- ADR-103/104/106/107 closure sweep: old non-active tails are now explicitly closed. ADR-103 keeps the stubbed interactive landing demo as completed scope and cancels/defer-indefinitely the public unauthenticated LLM demo endpoint; ADR-104 keeps D1/D2 as completed deploy-resilience scope and cancels/defer-indefinitely the auto-mutating drift-repair cron while leaving prod atomicity as a future design note; ADR-106 is marked completed; ADR-107 is marked closed as a partial program with its existing Program closure as source of truth.
- ADR-110 closed as completed after operator verification. The active execution order is finished: embedding truth moved to `Admin > Knowledge`, runtime text failover and retrieval-helper fallback execute on the native path, TTS model truth no longer depends on the old OpenAI hardcode, and Anthropic prompt-cache/accounting landed with the later stream-accounting hotfix. Deferred STT/web-search provider-model work stays out of ADR-110 and should return only as a separate future ADR/slice if product priority requires it.
- ADR-110 Anthropic stream-accounting hotfix: provider-gateway no longer double-counts Anthropic streaming usage by summing `message_start` and `message_delta` token snapshots as if they were additive deltas. Later stream usage payloads now replace earlier fields instead of accumulating them, which brings PersAI's Anthropic ledger/session token counts back in line with Anthropic's own logs and prevents false context-pressure / compaction signals on short chats.
- ADR-109 closed as completed after operator verification. The HeyGen talking-avatar foundation is accepted as done; remaining voice cloning, Characters UI v2, and video-player polish work is explicitly owned by ADR-111 and must not reopen ADR-109.
- ADR-111 opened as the orchestrator-run program for HeyGen voice cloning and Characters UX v2. It keeps ADR-109 closed, supersedes only ADR-109's deferred voice-cloning non-goal, excludes custom audio lip-sync / broad HeyGen passthrough tuning, and defines a 5-slice GPT-5.4 subagent execution plan covering video-player polish, premium Characters UI cleanup, cloned-voice substrate, HeyGen voice-clone backend/runtime wiring, voice-clone UI, docs, and live smoke.
- ADR-110 Anthropic prompt-cache fix: runtime now passes prompt-cache intent to Anthropic text calls instead of limiting prompt caching to OpenAI-only requests. Anthropic provider-gateway already projects that intent as `cache_control: { type: "ephemeral" }` on the stable system block, so live Claude calls can now produce `cacheCreationInputTokens` / `cachedInputTokens` in usage instead of always recording zero cache usage.
- ADR-110 live hotfix: Admin Runtime settings now tolerate persisted token-metered catalog rows created before `cacheCreationInputPer1M` existed by defaulting the missing cache-write price to `0` during normalization. The dev cluster's persisted runtime catalog was patched in place so existing OpenAI/Anthropic/image/video rows no longer break Admin Runtime reads or materialization rollouts.
- Billing lifecycle fix: CloudPayments recurrent `check` webhooks can now resolve an existing PersAI subscription by `SubscriptionId`/`AccountId` even when there is no local renewal payment intent, so provider retries are not rejected as an unknown payment intent. Manually/admin-activated paid subscriptions with no provider subscription reference now enter the same `active -> grace_period -> expired_fallback` lifecycle when their paid period expires, preserving the standard grace notifications, materialization rollout, and fallback behavior.
- ADR-110 Slice 4: Anthropic prompt cache is now provider-specific and economically honest. The Anthropic provider client keeps the stable `systemPrompt` cacheable via `cache_control: { type: "ephemeral" }` only when prompt caching is enabled, leaves volatile `developerInstructions` as an uncached suffix system block, and never emits a literal `developer` role to Anthropic. Usage/accounting now has a separate `cacheCreationInputTokens` bucket in addition to normal `inputTokens`, cache-read `cachedInputTokens`, and `outputTokens`; token pricing metadata adds `cacheCreationInputPer1M`, with Anthropic cache writes priced separately from ordinary input and cache reads in both quota weighting and the cost ledger.
- ADR-110 Slice 3: TTS keeps its existing runtime provider-chain fallback, but OpenAI TTS model truth is no longer hardcoded on the execution path. The API now materializes the OpenAI `text_to_speech` catalog entry onto the TTS credential ref, runtime/provider-gateway forward that `modelKey`, and OpenAI speech billing facts now reflect the actual successful model from the catalog-driven path. Yandex `speechkit-v3` remains intentional code-owned API-version truth in the active TTS path.
- ADR-110 scope correction: STT and web-search provider/model truth are deferred out of the active ADR-110 execution order because they are not current production blockers and need a later full vertical contract/runtime/provider-gateway/UI/test slice. The next active ADR-110 slice is now TTS model/default truth cleanup.
- ADR-110 Slice 2: runtime text failover now executes on the active runtime path for eligible text-model failures instead of stopping at the primary provider/model. `apps/runtime` now performs one-shot fallback for ordinary turn text calls plus router/background system-tool calls when the primary text request fails with timeout/5xx-style provider execution errors, and logs primary failure plus fallback success/failure explicitly. The API knowledge rerank helper keeps `retrievalModelKey = null` as graceful-off, but when configured it now retries once onto the runtime fallback provider/model for the same retryable helper failures. Embedding fallback, generated media, STT, TTS, and web-search behavior were left unchanged in this slice.
- ADR-110 Slice 1: embedding model truth moved out of Plans and into `Admin > Knowledge`. Plan payloads/contracts/UI no longer write `embeddingModelKey`; assistant-owned knowledge now resolves embedding/index/search model choice from `adminKnowledgeRetrievalPolicy`, preserving lexical graceful-off when admin embedding is unset or same-model embedding generation fails. Admin Knowledge policy updates now expose embedding-change impact (`affectedSourceCount`, chunks, bytes, by-source breakdown), require dangerous-action step-up confirmation when the embedding model changes, and enqueue backfill across all Admin Knowledge-governed sources: global uploaded Product KB, Product KB text entries, Skill documents, Skill knowledge cards, and assistant-uploaded knowledge. OpenAPI/generated contracts, admin-web tests, and focused API tests were updated to match the new source of truth.
- ADR-109 live delivery hotfix: three recent talking-avatar jobs in `persai-dev` were proven complete on HeyGen, but chat delivery still hung because HeyGen sometimes serves the finished `.mp4` with `Content-Type: binary/octet-stream`. The provider-gateway now normalizes empty / octet-stream HeyGen video downloads back to a real video MIME based on the download URL suffix (defaulting `.mp4` to `video/mp4`), so runtime no longer rejects completed HeyGen renders as “unsupported MIME type”.
- ADR-109 Characters edit flow: saved persona cards now open the same shared modal in `Edit` mode, but avatar replacement is intentionally forbidden there. Operators can update only the persona name and voice, preview the existing portrait in a larger view, and see a quiet hint that changing the face requires creating a new character. The API now exposes a bounded `PATCH /api/v1/workspaces/:workspaceId/video-personas/:personaId` path that updates only `displayName + heygenVoiceId`, preserving the existing `heygenAvatarId` and portrait identity instead of mutating HeyGen avatar state in place.
- ADR-109 live voice-catalog hotfix: HeyGen `GET /v3/voices` uses the opaque `next_token` response field, but the follow-up request must send it back as query parameter `token` per the official docs. The previous refresh path incorrectly sent `next_token=...`, and live dev verification from the `api` pod proved HeyGen kept returning the same first 20 EN voices with an unchanged cursor. The cache refresh now uses `token=...`, requests up to `100` rows per page, and stops early if HeyGen repeats the same cursor to avoid looping on a broken upstream page stream.
- ADR-109 live hotfix: fixed the Admin HeyGen voice refresh route so it goes through Clerk auth like the other admin tool-credentials endpoints; the live dev cluster root cause was a real `401 userId=null` on `POST /api/v1/admin/runtime/tool-credentials/heygen-voice-catalog/refresh`, which meant the spinner ran but the cache never refreshed.
- ADR-109 Admin Tools voice refresh feedback now shows `Last refresh ... · N voices` and a concrete post-refresh status like `Updated: 300 voices.` instead of a spinner-only transition with no visible outcome.
- ADR-109 HeyGen voice catalog truth-hardening: the shared HeyGen cache now follows provider pagination and stores the full deduped catalog, while runtime still materializes a short prompt-safe talking-avatar shortlist (`10 EN + 10 RU` target, balanced by gender as available) for model-facing instructions.
- ADR-109 Admin Tools now exposes a manual HeyGen voice refresh action with a visible last-refresh timestamp, so operators can force a catalog sync without waiting for the 24h cache TTL.
- ADR-109 Characters voice picker now reads the full HeyGen catalog instead of the model shortlist, and the `OTHER` bucket adds inline language search so non-EN/RU voices are browseable/listenable without bloating the runtime prompt.
- ADR-109 Characters UI polish: locked plans now show a single premium demo character with the provided demo portrait, a real voice preview, and a direct `https://persai.dev/app/pricing` upsell link; unlocked plans hide the demo card, move `Create character` below the saved list, add a quiet usage hint, and open saved portraits in a lightbox on click.
- ADR-109 voice-picker cleanup: fixed the RU/EN regression by normalizing human-readable HeyGen language labels like `English` / `Russian` into the same `en` / `ru` buckets used by the filter, and removed the nested-button markup in the create-persona voice list so the voice preview control no longer emits invalid HTML hydration warnings in tests.
- ADR-109 root-cause fix (saved-persona render): persona create/archive now best-effort mark workspace assistants `configDirtyAt` so the materialized spec re-builds and the model actually sees saved characters with their real `personaId` (previously the model saw an empty `videoPersonaCatalog`, told users "no saved characters", and improvised `personaId="alexey"` from the display name); and the persona repository `findById`/`archive` now guard against non-UUID ids and resolve to `null` (honest `persona_not_found`) instead of letting Prisma throw a raw "Error creating UUID" — that raw DB error was the cause of the "UUID error" the model paraphrased on saved-persona renders.
- ADR-109 fix (10-minute banner on a permanent error): the async media-job worker `assertVideoToolResultAccepted` threw `503 ServiceUnavailable` (retryable) for every `isError` video reason except `requested_mode_unsupported`, so permanent talking-avatar config/input failures (`persona_not_found`, `talking_avatar_persona_unavailable`, `talking_avatar_provider_unavailable`, `talking_avatar_plan_disabled`, `voice_not_found`, `voice_required`) were retried through the exponential-backoff budget (30s→60s→120s→240s→480s) for ~10–15 min before the banner cleared. These reasons are now classified non-retryable `400` and fail fast; genuinely transient reasons keep their `503`/retry budget.
